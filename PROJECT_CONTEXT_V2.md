## 1. Обзор

**QassaPos** — облачная SaaS кассовая система (POS) для точек фастфуда (рынок Узбекистана, валюта «сум»).

**Бизнес-модель:** владелец регистрируется → триал 14 дней (`status='trial'`) →
оплата вручную (Click/Payme/наличные) → администратор активирует тариф через
админ-панель (`/admin`, защита через `ADMIN_KEY`).

**Тарифы:** Старт ($29) · Бизнес ($49) · Сеть ($89, мульти-точки — **не реализован**).

**Функционал:** касса/чеки, меню+категории, склад в граммах, тех.карты
(автосписание ингредиентов при продаже), смены, клиенты+бонусы, сотрудники с
ролями (owner/admin/cashier), аналитика, управление подписками.

## 4. Модель данных (schema.sql)

```
accounts (id, name, email UNIQUE, password_hash, plan, status, expires_at, created_at)
  └─ venues (id, account_id→, name, currency='сум', bonus_pct=5)
       ├─ staff (id, venue_id→, name, phone, role, pin VARCHAR(4), active)
       ├─ menu_categories (id, venue_id→, name, sort_order)
       ├─ menu_items (id, venue_id→, category_id→, name, price INT, active)
       ├─ ingredients (id, venue_id→, name, stock_grams, min_grams, updated_at)
       ├─ recipes (id, menu_item_id→, ingredient_id→, grams)   -- тех.карты
       ├─ clients (id, venue_id→, name, email, phone, status, bonus, visits, total_spent, …)
       ├─ shifts (id, venue_id→, staff_id→, opened_at, closed_at, revenue)
       └─ orders (id, venue_id→, shift_id→, staff_id→, client_id→, total, pay_method, items_json JSONB)
```

- Мультиарендность: всё скоупится по `venue_id` (есть индексы `idx_*_venue`).
- Деньги — `INTEGER` (минимальные единицы, без копеек — для сума ок).
- Изоляция арендаторов соблюдается во всех роутах.

## 5. Аутентификация и авторизация

- **Владелец:** `POST /auth/login` (email+пароль) → JWT на **30 дней**.
- **Сотрудник:** `POST /auth/pin-login` (venueId+PIN) → JWT на **12 часов**.
- `requireAuth` проверяет токен И статус подписки аккаунта **на каждый запрос**
  (запрос к БД каждый раз). Возвращает `402 SUBSCRIPTION_EXPIRED` при истечении,
  `403` при `blocked`.
- `requireRole(...)` ограничивает действия по ролям.
- Админка: заголовок `x-admin-key` сравнивается с `process.env.ADMIN_KEY`.

**Переменные окружения:** `DATABASE_URL`, `JWT_SECRET` (обязателен, иначе
`process.exit(1)`), `ADMIN_KEY`, `CORS_ORIGIN` (опц.), `PORT`.

## 6. Результаты аудитов

Проведено два аудита (первичный обзор + верификация по строкам). Целевой
файл фронтенда — `backend/public/index.html`.

### 🔴 P0 — критично (безопасность)

**1. Stored XSS — уверенность 98%**
- Файл: `backend/public/index.html`.
- Данные из БД вставляются в `innerHTML` без экранирования; helper-экранирования
  нет. Точки внедрения: строки 764, 787, 879–887, 952, 1018–1029, 1092–1100,
  1116, 1148–1170, 1248–1255, 1308/1312/1321/1323.
- Вектор: имена клиентов/блюд/сотрудников/ингредиентов, в т.ч. через публичную
  регистрацию (`auth.js:14`). Токен и `ADMIN_KEY` в `localStorage` → кража.
- Усугубляется выключенным CSP (`server.js:16`).

**2. Хранение PIN в открытом виде — уверенность 100%**
- Файлы: `schema.sql:34` (`pin VARCHAR(4)` plaintext), `auth.js:43` (дефолт `0000`),
  `auth.js:109` (сравнение plaintext), `orders.js:211/219/230` (PIN в ответе API),
  `index.html:1251` (PIN показан в таблице).
- 4 цифры = 10000 комбинаций; rate-limit 80/15мин частично сдерживает перебор.

### 🟠 P1 — важно

**3. Дубликаты `index.html` — 100%.** `index.html` (root, 59 стр. расхождений),
`frontend/index.html` (242 стр.) — мёртвые копии; раздаётся только `public/`.

**4. Нет CSS для Kanban/клиентов — 100%.** В `backend/public/index.html` разметка
использует классы без правил в `<style>`: `client-tools`, `search-input`,
`view-switch`, `kanban-board`, `kanban-column`, `client-card*`, `receipt-*`
(строки 378–394, 1109–1125). Kanban и поиск отрисуются без стилей.

**5. Нет `.gitignore`** — риск закоммитить `.env`. Сейчас в гите только `.env.example`.

### 🟡 P2 — корректность/функционал

**6. «Выручка (всего)» по 50 заказам — 97%.** `orders.js:52` (`LIMIT 50`),
`index.html:1299–1303` суммирует ≤50 строк и подписывает «всего». То же —
средний чек, топ-блюд, статистика оплат.

**7. Часовой пояс сводки.** `orders.js:58` `today-summary` использует `CURRENT_DATE`
(UTC на Railway) вместо Asia/Tashkent (UTC+5) → граница «сегодня» смещена на 5 ч.

**8. Тариф «Сеть» не реализован.** Login берёт только первую точку
(`auth.js:70`), переключателя venue нет.

### Прочие наблюдения
- TOCTOU при открытии смены (`orders.js:20`) — возможны 2 открытые смены.
- Несовпадение статусов клиента и колонок Kanban (`getClientStatus` → «Редко» без колонки).
- PIN не уникален в точке → `pin-login` вернёт первого совпавшего.
- Чеки клиентов хранятся только в `localStorage` (base64) — не на сервере.
- Миграция `0001` не запускается автоматически (init.js гонит только `schema.sql`).
- CORS по умолчанию `origin:true` (отражает любой источник) при пустом `CORS_ORIGIN`.
- Запрос к БД в `requireAuth` на каждый вызов; N+1 в рецептах и создании заказа.
- README устарел (Шаг 7 «фронтенд не подключён» — неверно).

### ✅ Сильные стороны
- Все SQL-запросы параметризованы (SQL-инъекций нет).
- bcrypt для паролей; `JWT_SECRET` обязателен при старте.
- Последовательная изоляция по `venue_id`.
- Транзакция заказа со `SELECT … FOR UPDATE` по ингредиентам (защита от овэрселла).
- Валидация позиций заказа; rate-limit на auth; helmet подключён.
- Код чистый, читаемый, хорошо прокомментирован; малый объём.

## 9. Модуль «Печать чеков»

**Контекст:** backend в облаке, принтер — в локальной сети кассы → печать
инициирует браузер кассира, сервер только отдаёт данные чека. Целевое железо —
Android-планшеты / Windows mini-PC + термопринтер 58/80 мм (ESC/POS).

**Сравнение вариантов:**

| Критерий | Браузер `window.print` | QZ Tray | ESC/POS (Web Serial/USB) |
|---|---|---|---|
| Установка | ❌ нет | ⚠️ Java-агент | ⚠️ драйвер/доступ к порту |
| Android | ✅ | ❌ | ⚠️ нестабильно |
| Тихая печать | ⚠️ только Chrome kiosk | ✅ | ✅ |
| Денежный ящик | ❌ | ✅ | ✅ |
| Лицензия | ✅ бесплатно | ⚠️ платная подпись | ✅ бесплатно |
| Браузеры | ✅ все | ✅ (через агент) | ❌ только Chrome/Edge |
| Трудозатраты MVP | 🟢 0.5–1 день | 🟡 2–3 дня | 🔴 3–5 дней |

**★ Решение для старта: печать через браузер** — ноль установки, работает на
Android, бесплатно, закрывает 90% потребности. Заложить абстракцию
**PrintAdapter** (browser → позже QZ Tray / ESC/POS без переписывания).

**Архитектура:**
```
checkout() → buildReceiptModel(order, venue, cashier) → PrintAdapter.print(model)
                                                          ├ BrowserAdapter (MVP)
                                                          ├ QzTrayAdapter (фаза 2)
                                                          └ EscPosAdapter (фаза 2)
```
**MVP-план (~0.5–1 день):** миграция `print_*` в `venues` → эндпоинт данных чека
(`routes/orders.js`) → модель + адаптер + `printBrowser` (термо-CSS 58/80) →
кнопка/автопечать в `checkout()` → экран настроек печати.
**Риск:** экранирование данных чека (зависит от P0.1 XSS).
**Этап 1 можно вынести в Фазу 1–2** — частый запрос фастфуда, дёшево.
